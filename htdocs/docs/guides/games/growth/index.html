Social Channels for Games

Getting Started › Apps on Facebook › Social Channels for Games

Social Channels are a powerful tool for both increasing player engagement and new user acquisition. Take advantage of these tools for potentially exponential growth; if a user is enjoying your game make sure they have frequent and visible opportunities to invite their friends to play with them. These friends will then invite their friends and so on.

The first step is to become familiar with the different Social Channels available to you:

  • Requests - A transactional 1:1 communication between a sender and recipient.
  • Feed stories - Allows the developer to post unstructured stories that will appear in a user's Newsfeed and Timeline.
  • Open Graph - Allows the developer to post structured stories that will appear in Newsfeed, Ticker and Timeline Aggregations.
  • Scores & Achievements APIs - These enable developers to publish their users' high scores and achievements for their app. These are built on Open Graph and get distribution via the same channels as other Open Graph actions.
  • Bookmarks - Once a user starts playing your game, we create a bookmark to enable users to easily navigate back to your game from within Facebook. These Bookmarks are available on the Facebook homepage as well as on the Canvas Page on the top right corner. Developers get these for free and do not have to do anything special to get access to this channel.

Use the matrix below to compare the Social Channels and select the appropriate Social Channel for your use case. The terms are defined as:

  • User Interaction - the direction of communication: sender to single recipient or multiple recipients.
  • Visibility - who can view the communication: only the recipient or everyone.
  • Value - when to use the channel: acquiring new users or engaging existing users.
  • State - whether the communication is permanent or transient.

For more information on implementation details for our Social Channels please see the Canvas Tutorial and the Social channels docs. Next let’s focus on how Games Developers can use these channels specifically for Social Games by walking through some examples and best practices.

Requests

Requests are a 1:1 communication channel between a sender and recipient. Requests are always private and can only be seen by the recipient. Requests follow a transaction model - the sender is requesting some action from the recipient, that the recipient can accept or not.

Requests can be sent in 3 scenarios:

  1. The recipient is a friend of the sender and has not installed the game. This is considered an invite.
  2. The recipient is is a friend of the sender and has installed the game.
  3. The recipient has installed the game. In this case the sender and recipient do not need to be friends. This is extremely valuable for scenarios like match making discussed below.

Feed Stories

The Feed stories represents the users Newsfeed and Timeline. Stories posted via the Feed Dialog will appear in the users friends' Newsfeed and ultimately post to their Tieline, so long as they are both engaged with the game. These unstructured stories will not appear in Ticker. Feed Stories can be either one to many - the user is posting to their own Timeline or one to one - the user is posting to a friends Timeline.

Feed Stories contain unstructured data, essentially strings, that are customized by the developer. One notable feature of Feed Stories that differentiates it from the other Social Channels is the ability for a user to enter a custom message. This results in a more personal story and can lead to higher engagement from the user's friends.

Next we'll discuss a core scenario for feed stories - Bragging.

Bragging Stories

A common use-case for Feed are “Brag” stories. These are intentional posts made on the users behalf when they have completed an activity of note within the game; anything from completing a quest, earning a new level or defeating a player. Bragging is a great use case that tends to drive a high level of engagement due to the competitive nature of games.

Pro-Tip: As mentioned previously, one of the key features that differentiate Feed is the ability for a user to add a message to the Story. This results in a more personal story and can lead to higher engagement from the user's friends. Try to limit using Feed stories to interesting and note worthy moments within your game; these should be intentional shares that the user is invested in making it more likely for them to include a personal message.

For more detailed information, check out the Feed Dialog docs.

Open Graph

Facebook has recently updated the Open Graph to support built-in and custom objects and actions. This feature is currently in beta and available to all registered Developers.

One of biggest differences between Open Graph and Feed is the ability to publish structured data. All stories must follow a "actor verb noun" syntax, such as "Constantin conquered a universe". By using structured data developers can create Timeline Aggregations that roll up these stories to highlight interesting patterns which become a permanent part of the user's identity on Facebook. These stories get distribution in the user's Newsfeed, Ticker as well as Timeline. In addition with Open Graph developers can provide translated text for their objects and actions in different languages enabling us to display these stories to viewers in their preferred languages resulting in increased engagement with the stories.

For an overview of the new features available, please check out our docs. A great use case for both Feed and Open Graph is enabling apps to publish user generated content. We will also leverage this use case to highlight the differences between the two.

User Generated Content

A great use-case for the social channels is to distribute user generated content. Let’s consider a cooking game where a user can construct and decorate a dish. The player has spent some time crafting a unique dish that they now want to show off. Developers can use both Feed stories and Open Graph to show case and broadcast the results of the players efforts to their friends.

A Feed story can be used to include a custom image of the users dish and player submitted text describing their accomplishment. This results in the dish being shown to their friends and allowing the viewers to comment and like the post, helping to drive engagement and reward the player for their actions. This may get distribution to the player's friends NewsFeeds and will show up on the player's Timeline.

Similarly Open Graph can be used to share the dish, by creating a custom action Cook and a custom object Dish, which may be displayed on the player's friends' Ticker and News Feed and creates the opportunity for some interesting developer created Aggregations on the players Timeline.

The main distinctions to these two approaches are:

Feed Stories

  • These are unstructured.
  • They can contain any content the developer specifies, along with custom text entered by the player.
  • They should always be intentional where the user explicitly chooses to publish something and can control the directed audience.
  • Feed stories will show up on the user's Timeline but are transient and will not be permanent. The story will receive distribution in the Newsfeed and is viewable by users of the game that posted the story.

Open Graph

  • Open Graph stories are structured and must conform to the “actor verb object” model.
  • They are non-interruptive to the user's game play and allow the developer to create a live channel on Facebook for the most meaningful events happening in their game.
  • The story will receive distribution through Ticker, Newsfeed and Timeline.
  • Timeline Aggregations can be used to aggregate the data into interesting aggregations which live permanently on the user's Timeline.

Both approaches are valid and the developer should weigh each option against their desired goals.

Scores and Achievements

Scores and achievements APIs are built on Open Graph and receive the same distribution points that all Open Graph stories do - Ticker, Newsfeed and Timeline. In addition to these scores and achievements also benefit from built in aggregations and stories that have been designed by Facebook.

Most games already use some form of scores and achievements in their games. These APIs give developers the ability to easily publish stories when users achieve high scores and achievements in their games, to drive engagement for their users and acquire new users.

Scores

The Scores API can be used to submit scores for any games that supports scores and/or a leaderboard.

When a user earns a new high score, a high score story is generated and displayed automatically in Feed, Canvas Ticker and the user's Timeline. Facebook also provides leaderboard flyouts for the scores landing stories as they appear in Ticker.

Pro-Tip: Developers should supplement scores by also providing their own leaderboard that persists within the game and not rely solely on the scores stories displayed via the Social Channels. The persistent leaderboard encourages users to constantly compete with friends and improve on their scores.

Achievements

Achievements API can be used to define a set of achievements for your game and then award them to users once the criteria have been met. Games have a 1000 point allocation for all achievements and each achievement object must be allocated at least 1 point. achievements points are used to describe the value of the achievement in relation to all achievements defined in the game. The different achievement point values result in the following distribution effects:

  • 1-9 points – no distribution, though the achievement will display in Timeline Aggregations.
  • 10 – 24 points – Ticker story with no image.
  • 25 – 49 points – Ticker and Feed story with image.
  • 50 – 99 points – Ticker and Feed story with image and additional distribution.
  • 100+ points – Same as above with even MOAR distribution.

Regardless of the point allocation, all Achievements will be display in Timeline Aggregations.

Pro-Tip: When designing Achievements consider the target audience. A key difference with Achievements on Facebook Platform is that both users and non users of the game will see these stories.

If the developer is targeting distribution, design around making the achievements as accessible to non users as possible. Creating an achievement that is so insular to the game that only hard core players would understand will likely receive very little click through from non users and be less successful as a distribution mechanic.

For developers targeting engagement, these guidelines are not necessarily as strict. In this case players may bond around a very specific and difficult to get achievement, so long as the criteria to achieve it widely known.

Next steps

To get started, please familiarize yourself with the following technologies as, in most cases developers will need to interact with them:

  • JavaScript SDK - a thin wrapper on the Dialog API and Graph API, used to display UI elements for Facebook apps.
  • Graph API - a RESTful API for interacting with the Social Graph.
  • Credits API - an API to implement payments with the Facebook Platform. This is required for Canvas games.
