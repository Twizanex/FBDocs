Authentication Pro-tips

Getting Started › Apps on Facebook › Authentication Pro-tips

Authentication gives your game the ability to know the identity of a Facebook user, and to read and write data via Facebook's APIs. To make your Authentication Dialog user-friendly, only ask users for permissions that you need, and use a quality logo and description that get users interested in playing your game.

1. Only ask for the permissions that you need.

When asking for permissions, consider the user experience and impact to conversion. Don’t ask for permissions that you don’t directly need to provide the base user experience. We’ve seen that the conversion rate for the Auth Dialog declines proportionally to the number of permissions a developer requests when a new user installs the game.

Developers can always ask for additional permissions later. When asking for these permissions only do so when it is beneficial for the user experience and make sure to be clear as to how you will be using them. For example, publish_stream generally isn’t required for most games to function. However, if you notice that a player is frequently sharing Feed posts consider asking them if they would like to grant publish_stream for a more seamless experience. The player can revoke the permission at any time if they no longer find value in it via editing their app privacy settings. For this reason if you are using extended permissions make sure to verify what permissions you have access to when the user logs in.

To retrieve what permissions a current user has granted use the Graph API and request the permissions connection.

To prompt a user for additional permissions use FB.login via the JavaScript SDK.

2. Email and Publish actions Permissions

By default, when authorizing your application, a user grants your app access to their basic information. Two additional permissions that game developers frequently request are the email permission, and publish_actions.

  • Email permission

The email permission provides access to the user's primary email address, so that you can re-engage your users outside of Facebook if you want to let them know about things like new content in your game and upcoming promotions. (Note: Do not spam users. Your use of email must comply both with Facebook policies and with the CAN-SPAM Act.)

  • Publish_actions permission

In order to start publishing scores and achievements, a user needs to grant your app the publish_actions permission. The publish_actions permission also gives you the ability to publish feed stories in a structured way, without popping a dialog each time.

Angry Birds asks for email and publish_actions permissions

These common permissions are displayed in a more condensed, bucketed form to users, and do not create a second Auth Dialog screen. However, if you request any other extended permissions, your users will have to go through a second dialog page after installing your application. We have found that requesting permissions that prompt a second Auth Dialog screen can lead to lower conversion rates. Read more in our Publishing Permissions Guide.

Note: Many developers have traditionally used the publish_stream permission. The publish_stream permission is a superset of publish_actions permissions, and you only need this if you need the ability to post to a friend’s feed or post content to events or groups. Since publish_stream is a second screen Auth Dialog permission, make sure you really need the specific capabilities solely offered through publish_stream before requesting it. ￼￼

3. Use a great logo and description for your Auth Dialog.

In the Customize section of the Auth Configuration, you should update how your Auth Dialog will appear to users. In particular, your app logo matters. We recommend using a game character or other engaging image that will resonate with your users.

4. Track Auth dialog conversion for new users

You can track how well your Auth Dialog is converting by checking out insights for your app under the "Installs" tab.

You will be able to see overall conversion numbers for your auth dialog as well as broken down by the different set of permissions you are experimenting with. Use this data to optimize the set of permissions you ask for your app upfront as well as the logo and the description for your dialog.
